# DOR-142 — Ingest Validation

## Ticket

**Summary:** Integrity check when changes are committed to the repository

**Description:**
> When changes are committed to the repository, we can ensure that the digests
> calculated by the repository (OCFL) match what was supplied in the package.

---

## Context and Motivation

When a depositor submits a package to DOR Depot, the intake pipeline validates
each content file against its SHA-512 digest recorded in the `.dor/` sidecar header
files. Those headers travel into the OCFL object as content files alongside the data.
The OCFL repository then independently computes and records digests in its own
`inventory.json` manifest.

This creates two independent digest records for every ingested file:

| Record           | Where stored                                                  | Computed by                   |
|------------------|---------------------------------------------------------------|-------------------------------|
| **Inbox digest** | `.dor/<filename>.json` → `digests[0]` field (`urn:sha-512:…`) | Depositor / intake validation |
| **OCFL digest**  | `inventory.json` → `manifest` block                           | OCFL library at commit time   |

The ingest validation check cross-references these two records. If they match,
there is a cryptographic proof that _the bits the depositor supplied are exactly
the bits the repository stored_ — independent of whether the storage pipeline is
trusted. This is distinct from the standard OCFL fixity check (which only
guarantees internal consistency within the repository).

This external-consistency guarantee is socially and legally essential for digital
preservation: it allows the Depot to tell depositors "the bits you gave us are
the bits we have."

## Design

### Trigger

`POST /admin/ingest-validation?objectId=<binId>&agentName=<name>&agentAddress=<address>`

This publishes `PerformIngestValidation(BinId, Agent)` via `AdminController` →
`AdminService`.

### Handler

`IntegrityService.on(PerformIngestValidation)` → `performIngestValidation(command)`

### Algorithm

For a given `binId`:

1. **Verify object exists** in the OCFL repository via `preservationGateway.hasObject()`.
   If not, throw `PreservationNotFoundException` → caught by the `on()` listener →
   publish `IngestValidationTargetMissing`.

2. **Read all `.dor/` header sidecar files** for the object from OCFL (all files
   whose logical path ends with `.dor/<name>.json`). Each sidecar contains:
   - `contentPath` — logical path of the corresponding content file
   - `digests` — array of `urn:sha-512:<hex>` URNs; the first entry is the
     original digest validated at intake time

3. **Fetch all manifest digests** for the object via
   `preservationGateway.getContentFileDigests(binId)`, which returns a
   `Map<String, String>` of logical path → SHA-512 hex. Internally this calls
   `repo.describeObject(id)` and reads `FileDetails.getFixity().get(DigestAlgorithmRegistry.sha512)`
   for each file in the head version — no `inventory.json` stream parsing required.

4. **Cross-check each sidecar against the manifest**:
   - Read the sidecar JSON via `getFileStream` and extract the inbox digest with
     `extractDigest()`, which handles both sidecar formats:
     - Curio (`MinimalHeader`): top-level `"digest": "hex..."` field
     - Glam (`Header`): `"digests": ["urn:sha-512:hex..."]` array
   - Derive the content file's logical path from the sidecar path
     (`<parent>/.dor/<name>.json` → `<parent>/<name>`).
   - Look up that logical path in the manifest digest map.
   - Compare the two digests (case-insensitive).

5. **Record a PREMIS `VALIDATION` event** to both the OCFL log files and the
   database (`PreservationEventsStorage` + `PreservationEventMapper`), with:
   - `type`: `PreservationEventType.VALIDATION`
   - `outcome`: `SUCCESS` or `FAILURE`
   - `outcomeDetail`: human-readable summary of which files matched/failed
   - agents: executing system (`AgentRole.EXE`) and requesting agent (`AgentRole.REQ`)

6. **Publish outcome event**:
   - All files match → `IngestValidationPassed(binId)`
   - Any discrepancy → `IngestValidationDiscrepancyDetected(binId)`

### Events and Classes

| Class                                         | Role                                                  |
|-----------------------------------------------|-------------------------------------------------------|
| `PerformIngestValidation`                     | Command record (trigger)                              |
| `IntegrityService`                            | Event listener and validation logic                   |
| `IngestValidationPassed`                      | Outcome event — all digests matched                   |
| `IngestValidationDiscrepancyDetected`         | Outcome event — one or more mismatches                |
| `IngestValidationTargetMissing`               | Outcome event — object not found                      |
| `PreservationEventType.VALIDATION`            | PREMIS event type written to log + DB                 |
| `PreservationGateway.getContentFileDigests()` | Returns manifest digests (logical path → SHA-512 hex) |

All items are complete. PR open on `DOR-142/ingest-validation`.

- `PerformIngestValidation`, `IngestValidationPassed`, `IngestValidationDiscrepancyDetected`,
  and `IngestValidationTargetMissing` records exist

- Three test scenarios pass: clean ingest, sidecar tampered, object not found

## Notes

- Header files may be modified during intake (additional metadata may be written).
  The validation must use the `digests` array value — not recompute from the file
  on disk in the intake area — because the inbox may no longer exist by the time
  validation runs.
- The check targets committed OCFL objects. Staged (mutable) objects are not a
  target for this validation.
- The AMQP `integrity.exchange` / `integrity.queue` is already configured in
  `AmqpIntegrationConfiguration` and can carry `PerformIngestValidation` events
  externally if needed.
